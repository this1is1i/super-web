# AGENTS.md

This file provides guidance to Codex (Codex.ai/code) when working with code in this repository.

## What this is

超级井字棋网络对战版 (Super Tic-Tac-Toe, network multiplayer). Vanilla-JS browser client + a Node.js WebSocket server. No framework, no bundler, no build step, no tests. The only dependency is `ws`.

## Running it

```bash
npm install                 # installs ws
npm start                   # HTTP frontend + WebSocket server on port 8080
npm test                    # shared-rule and two-client protocol tests
```

The HTTP and WebSocket services share the same port. The browser automatically connects back to the host that served the page.

## Architecture

Two peers exchange JSON messages over one WebSocket connection, mediated by a room.

- **`js/server.js`** — authoritative HTTP/WebSocket server. Validates room membership, turns and legal cells, then applies moves through the shared rules module. It also owns swap-request confirmation and cooldown state.
- **`js/main.js`** — browser client. Holds its own copy of `gameState` and applies **optimistic updates** through the shared rules module. If the server replies `error`, it rolls back using `pendingMoveSnapshot`.
- **`js/game-rules.js`** — UMD-style pure game rules shared by Node.js and the browser: move resolution, board routing, winning-pattern capture, bonus calculation and game-end scoring.
- **`index.html`** — single page; buttons call global functions in `main.js` directly (`connectToServer()`, `createRoom()`, `joinRoom()`, `resetGame()`, etc.). The 9 mini-boards are generated by `renderBoard()`.
- **`css/main.css`** — all styling.

### CRITICAL: game logic must stay shared

Both server and client must call `js/game-rules.js` for state transitions. Do not recreate move-resolution or scoring logic inside `server.js` or `main.js`; duplicated rules will make optimistic prediction diverge from the authoritative result.

### Game rule specifics (the non-obvious parts)

- Play starts on the center board (index `4`).
- Normally, the cell you play (`cell_index`) dictates which board the opponent must play next.
- Each board records `fromBoard` — the board that last redirected play into it — used to backtrack when the normal target board is already won/full.
- When a board is won or drawn, `findNextBoardAfterWin` picks the next board: prefer the played cell's board, else the `fromBoard` chain (`findRecursiveAvailableBoard` walks it), else the first available board.
- Scoring: +1 per mini-board won. An overall 3-in-a-row gives +2, or +3 when all three boards share at least one winning pattern without forming the corresponding physical long line. Each complete same-color physical 9-cell row, column or long diagonal gives +4. When both conditions describe the same achievement, only the +4 long-line bonus applies. Independent bonuses still stack and are recomputed after every move.

### WebSocket message protocol

Client → server: `create_room`, `join_room` (`room_id`), `make_move` (`room_id`, `move{board_index, cell_index, player}`), `request_swap`, `respond_swap`, `reset_game`, `leave_room`.

Server → client: `room_created`, `game_start`, `move_made`, `swap_request_sent`, `swap_request`, `swap_result`, `swap_unavailable`, `game_reset`, `game_over`, `error`, `player_disconnected`.

- `leave_room` — sent when a player cancels waiting or voluntarily leaves; server removes the player and cleans up the room. Replaces the previously documented-but-unimplemented `cancel_waiting`.
- `game_over` — broadcast after the final move when all 9 boards are decided; includes `winner` and `scores`.
