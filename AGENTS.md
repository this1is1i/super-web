# AGENTS.md

This file provides guidance to Codex (Codex.ai/code) when working with code in this repository.

## What this is

超级井字棋 (Super Tic-Tac-Toe) with local AI and network multiplayer. Vanilla-JS browser client + a Node.js WebSocket server. No framework, bundler, or build step. The only runtime dependency is `ws`; tests use Node's built-in test runner.

## Running it

```bash
npm install                 # installs ws
npm start                   # HTTP frontend + WebSocket server on port 8080
npm test                    # shared-rule and two-client protocol tests
```

The HTTP and WebSocket services share the same port. The browser automatically connects back to the host that served the page.

## Architecture

Online peers exchange JSON messages over WebSocket connections mediated by a room; AI games stay local in the browser.

- **`js/server.js`** — authoritative HTTP/WebSocket server. Validates room membership, state versions and turns, chooses chaos exchanges, then calls the shared rules module. It also owns chat history, five-minute reconnect sessions, and pre-game swap confirmation/cooldown.
- **`js/main.js`** — browser client. Online moves remain visually pending until the server broadcasts `turn_applied`; local AI games call the shared rules directly.
- **`js/game-rules.js`** — UMD-style pure game rules shared by Node.js and the browser: atomic turn resolution, movable mini-board mapping, routing, winning patterns, bonus calculation and game-end scoring.
- **`js/ai-player.js`** — local normal heuristic and bounded hard minimax AI. It simulates only through `game-rules.js`.
- **`index.html`** — single page; buttons call global functions in `main.js` directly (`connectToServer()`, `createRoom()`, `joinRoom()`, `resetGame()`, etc.). The 9 mini-boards are generated by `renderBoard()`.
- **`css/main.css`** — fixed-viewport single-screen layout. The central board is responsive; rules use a modal and connection/room controls use an overlay drawer.

### CRITICAL: game logic must stay shared

The server, client and AI must call `js/game-rules.js` for state transitions. Do not recreate move resolution, exchanges or scoring elsewhere.

### Game rule specifics (the non-obvious parts)

- Play starts on the center board (index `4`).
- The current board's orange `active` highlight is visible to both players; the separate `playable` class is local-turn-only and controls hover/click affordances.
- Two independent settings form nine combinations: board variant `normal|cycle|chaos` and opponent `pvp|ai_normal|ai_hard`. Defaults are `normal+pvp`; humans are X and move first in AI games.
- Absolute positions stay numbered 0–8. Mini-board contents are tiles whose identities move through `positionToTile`; `fromTileId` follows tile identity and is resolved back to its live absolute position.
- Normally, the cell you play (`cell_index`) dictates the absolute position the opponent must play next.
- Cycle exchanges use `0-1, 1-2, ... 7-8, 8-0`; chaos exchanges use one server-authoritative random pair. An exchange happens every `swapEvery` successful turns (1–20).
- Atomic order is: settle the mark, exchange if due, resolve the next absolute position on the new mapping, then recompute bonuses and game end. Won/drawn tiles may also exchange.
- When the direct target is unavailable, routing follows the `fromTileId` chain at its current position, then falls back to the first playable position.
- Scoring: +1 per mini-board won. An overall 3-in-a-row gives +2, or +3 when all three boards share at least one winning pattern without forming the corresponding physical long line. Each complete same-color physical 9-cell row, column or long diagonal gives +4. When both conditions describe the same achievement, only the +4 long-line bonus applies. Independent bonuses still stack and are recomputed after every move.

### WebSocket message protocol

Client → server: `create_room` (`rule_config{boardVariant,swapEvery}`), `join_room`, `make_move` (`room_id`, `client_move_id`, `state_version`, `move{position,cell_index}`), `chat_send`, `resume_session`, `confirm_resume`, `request_swap`, `respond_swap`, `reset_game`, `leave_room`.

Server → client: `room_created`, `game_start`, `turn_applied`, `chat_message`, `chat_error`, `session_resumed`, `resume_confirmed`, `session_error`, `player_temporarily_disconnected`, `player_reconnected`, `player_disconnected`, swap events, `game_reset`, `error`.

- `turn_applied` is the only online move success event. It includes the authoritative turn (including the selected exchange pair), result and incremented `state_version`.
- `client_move_id` is idempotent within a per-player 256-entry LRU window.
- Text and image chat share the room connection. Room history keeps the newest 100 messages within 10 MiB; supported images are JPEG/PNG/WebP up to 512 KiB.
- Unexpected disconnects reserve the seat for five minutes. `leave_room` bypasses the grace period and destroys the room immediately.
