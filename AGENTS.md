# AGENTS.md

This file provides guidance to Codex (Codex.ai/code) when working with code in this repository.

## What this is

超级井字棋网络对战版 (Super Tic-Tac-Toe, network multiplayer). Vanilla-JS browser client + a Node.js WebSocket server. No framework, no bundler, no build step, no tests. The only dependency is `ws`.

## Running it

```bash
npm install                 # installs ws
node js/server.js           # WebSocket game server on port 8080
```

The server (`js/server.js`) only handles WebSocket connections — it does **not** serve the static frontend (`http.createServer()` has no request handler). `index.html` references assets with absolute paths (`/css/main.css`, `/js/main.js`), so to run the client you need a static HTTP server rooted at the **project root**, e.g. `npx serve .` or `python -m http.server`. Then connect the client to `ws://localhost:8080` (the default value in the connect input).

There are no lint or test commands configured.

## Architecture

Two peers exchange JSON messages over one WebSocket connection, mediated by a room.

- **`js/server.js`** — authoritative game server. Holds `rooms` (a `Map` of roomId → `{ players[], gameState }`). Validates every move (correct turn, legal cell, matches `currentBoard`), mutates the room's `gameState`, then broadcasts `move_made` to both players. Handles `create_room` / `join_room` / `make_move` / `reset_game` and cleans up rooms on `close`.
- **`js/main.js`** — browser client. Holds its own copy of `gameState` and applies **optimistic updates**: on click it sends the move AND immediately applies it locally. If the server replies `error`, it rolls back using `pendingMoveSnapshot`. `handleOpponentMove` applies the other player's confirmed move.
- **`index.html`** — single page; buttons call global functions in `main.js` directly (`connectToServer()`, `createRoom()`, `joinRoom()`, `resetGame()`, etc.). The 9 mini-boards are generated by `renderBoard()`.
- **`css/main.css`** — all styling.

### CRITICAL: game logic is duplicated in two places

The move-resolution rules are implemented **twice** — once in `js/server.js` and once in `js/main.js` — as near-identical copies of these functions:

`checkMiniBoardWin`, `checkMiniBoardDraw`, `findNextBoardAfterWin`, `findRecursiveAvailableBoard`, `checkOverallWin`, `checkGameEnd`.

**Any change to game rules must be mirrored in both files**, or the client's optimistic prediction will diverge from the authoritative server state and cause rollbacks or desync. The server copy is authoritative; the client copy exists only to predict the outcome locally.

### Game rule specifics (the non-obvious parts)

- Play starts on the center board (index `4`).
- Normally, the cell you play (`cell_index`) dictates which board the opponent must play next.
- Each board records `fromBoard` — the board that last redirected play into it — used to backtrack when the normal target board is already won/full.
- When a board is won or drawn, `findNextBoardAfterWin` picks the next board: prefer the played cell's board, else the `fromBoard` chain (`findRecursiveAvailableBoard` walks it), else the first available board.
- Scoring: +1 per mini-board won; +2 bonus for making a 3-in-a-row of won boards on the overall board (`checkOverallWin`, recomputed each move). Game ends when all 9 boards are decided; higher total (score + bonus) wins.

### WebSocket message protocol

Client → server: `create_room`, `join_room` (`room_id`), `make_move` (`room_id`, `move{board_index, cell_index, player}`), `reset_game` (`room_id`), `leave_room` (`room_id`).

Server → client: `room_created`, `game_start`, `move_made`, `game_reset`, `game_over`, `error`, `player_disconnected`.

- `leave_room` — sent when a player cancels waiting or voluntarily leaves; server removes the player and cleans up the room. Replaces the previously documented-but-unimplemented `cancel_waiting`.
- `game_over` — broadcast after the final move when all 9 boards are decided; includes `winner` and `scores`.
